# Architecture Decision Records (ADR)

## ADR-001: Python을 주 개발 언어로 선택

### 상태
승인됨 (2024-01-15)

### 컨텍스트
데이터 플랫폼 개발을 위한 프로그래밍 언어 선택이 필요합니다.

### 결정
Python 3.11+을 주 개발 언어로 사용합니다.

### 근거
**장점:**
- 데이터 처리에 최적화된 생태계 (Pandas, NumPy, Polars)
- 풍부한 라이브러리 (FastAPI, SQLAlchemy, Pydantic)
- 빠른 개발 속도 및 높은 생산성
- 데이터 엔지니어/과학자들에게 친숙한 언어
- 타입 힌트로 코드 안정성 확보

**단점:**
- 실행 속도가 상대적으로 느림
- GIL로 인한 멀티스레딩 제약

**대안:**
- Go: 성능은 우수하나 데이터 처리 생태계 부족
- Java: 엔터프라이즈급이나 개발 속도 느림
- Rust: 최고 성능이나 학습 곡선 가파름

### 결과
- 빠른 MVP 개발 가능
- 데이터 처리 작업에 최적화
- 팀원들의 빠른 온보딩

## ADR-002: FastAPI를 REST API 프레임워크로 선택

### 상태
승인됨 (2024-01-15)

### 컨텍스트
REST API 서버 구축을 위한 프레임워크 선택이 필요합니다.

### 결정
FastAPI를 사용합니다.

### 근거
**장점:**
- 자동 API 문서 생성 (OpenAPI/Swagger)
- Pydantic 기반 데이터 검증
- 비동기 처리 지원 (async/await)
- 높은 성능 (Starlette 기반)
- 타입 힌트 기반 개발

**단점:**
- 상대적으로 새로운 프레임워크
- 일부 고급 기능은 직접 구현 필요

**대안:**
- Flask: 성숙하나 비동기 지원 부족
- Django: 무겁고 불필요한 기능 많음
- Express.js: Node.js 생태계 (Python 아님)

### 결과
- API 개발 속도 향상
- 자동 문서화로 유지보수 용이
- 높은 처리 성능

## ADR-003: PostgreSQL을 주 데이터베이스로 선택

### 상태
승인됨 (2024-01-16)

### 컨텍스트
구조화된 데이터 저장을 위한 데이터베이스 선택이 필요합니다.

### 결정
PostgreSQL 15+를 사용합니다.

### 근거
**장점:**
- 강력한 ACID 보장
- 풍부한 데이터 타입 (JSONB, Array, etc.)
- 파티셔닝 지원 (날짜 기반)
- 우수한 인덱싱 (B-tree, GiST, GIN)
- 확장성 (Foreign Data Wrapper)
- 무료 오픈소스

**단점:**
- 수평 확장이 어려움 (샤딩 복잡)
- 대용량 쓰기 시 성능 저하 가능

**대안:**
- MySQL: 기능이 제한적
- MongoDB: 스키마리스이나 ACID 보장 약함
- Cassandra: 분산 DB이나 복잡도 높음

### 결과
- 데이터 무결성 보장
- 복잡한 쿼리 지원
- 안정적인 운영

## ADR-004: Pandas를 데이터 처리 라이브러리로 선택

### 상태
승인됨 (2024-01-16)

### 컨텍스트
대용량 데이터 처리를 위한 라이브러리 선택이 필요합니다.

### 결정
Pandas를 주 데이터 처리 라이브러리로 사용합니다.

### 근거
**장점:**
- 데이터 처리의 사실상 표준
- 풍부한 기능 (필터링, 집계, 변환)
- 다양한 포맷 지원 (CSV, JSON, Parquet, SQL)
- 대규모 커뮤니티 및 문서
- NumPy 기반 고성능

**단점:**
- 메모리 사용량이 큼
- 매우 큰 데이터셋 처리 제한

**대안:**
- Polars: 더 빠르나 생태계 작음
- Dask: 분산 처리 가능하나 복잡도 증가
- PySpark: 대용량 처리 가능하나 오버헤드 큼

### 결과
- 빠른 개발 및 디버깅
- 풍부한 기능 활용
- 팀원들의 높은 숙련도

## ADR-006: Redis를 캐싱 레이어로 선택

### 상태
승인됨 (2024-01-17)

### 컨텍스트
자주 조회되는 데이터의 성능 향상을 위한 캐싱이 필요합니다.

### 결정
Redis 7+를 캐싱 레이어로 사용합니다.

### 근거
**장점:**
- 매우 빠른 읽기/쓰기 (in-memory)
- 다양한 데이터 구조 (String, Hash, List, Set)
- TTL 지원

- 간단한 운영

**단점:**
- 메모리 제약
- 데이터 영속성 제한

**대안:**
- Memcached: 기능 제한적
- Hazelcast: 복잡도 높음
- 로컬 캐시: 분산 환경 지원 불가

### 결과
- API 응답 시간 단축
- 데이터베이스 부하 감소
- 확장 가능한 캐싱 전략

## ADR-007: Parquet을 데이터 레이크 포맷으로 선택

### 상태
승인됨 (2024-01-18)

### 컨텍스트
대용량 데이터의 장기 보관을 위한 파일 포맷 선택이 필요합니다.

### 결정
Apache Parquet 포맷을 사용합니다.

### 근거
**장점:**
- 컬럼 기반 저장 (분석 쿼리 최적화)
- 높은 압축률 (70-80%)
- 스키마 포함 (self-describing)
- 다양한 도구 지원 (Spark, Pandas, DuckDB)
- 파티셔닝 지원

**단점:**
- 행 단위 읽기 느림
- 파일 수정 불가 (immutable)

**대안:**
- CSV: 압축률 낮고 타입 정보 없음
- JSON: 저장 공간 비효율적
- Avro: 행 기반 (분석에 비효율적)

### 결과
- 저장 공간 절약
- 빠른 분석 쿼리
- 표준 포맷 사용

## ADR-012: Black + Pylint를 코드 품질 도구로 선택

### 상태
승인됨 (2024-01-20)

### 컨텍스트
일관된 코드 스타일 및 품질 유지가 필요합니다.

### 결정
Black (formatter) + Pylint (linter)를 사용합니다.

### 근거
**장점:**
- Black: 논쟁 없는 포맷팅 (opinionated)
- Pylint: 포괄적인 코드 검사
- 자동화 가능 (pre-commit hook)
- IDE 통합 용이

**단점:**
- Black의 엄격한 규칙 (일부 불편)

**대안:**
- Flake8: 기능 제한적
- autopep8: 덜 엄격함
- Ruff: 빠르나 아직 성숙도 낮음

### 결과
- 일관된 코드 스타일
- 코드 리뷰 시간 단축
- 버그 조기 발견
